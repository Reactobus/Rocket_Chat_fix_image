# Инструкция: как патчить Rocket.Chat (Docker)

Документ описывает порядок действий при доработках серверного бандла (**`app.js`**) и пересборке образа. Цель — не смешивать версии и иметь повторяемый контур для следующих апдейтов.

## Статус проверки (прод, 2026-04)

Связка **Rocket.Chat 8.3.2** + **`patch_appjs_upload_names.py`** (три якорных вставки: **`uploadsOnValidate`**, после **`Uploads.updateFileComplete`** в **`sendFileMessage`**, visitor **livechat**) подтверждена: в интерфейсе (в т.ч. мобильный просмотр вложения) отображается **нормализованное** имя файла — транслитерация кириллицы, пробелы и недопустимые символы заменены на **`-`**. Сообщения, созданные **до** выкладки патча, в MongoDB по-прежнему хранят старый **`title`**; для проверки используйте **новую** отправку файла с «грязным» именем.

## 1. Принципы

| Правило | Зачем |
|--------|--------|
| **`app.js` обязан соответствовать тегу базового образа** | Бандл — это скомпилированное целое под конкретный релиз; чужой `app.js` на другом теге даст сбои или тихие баги. |
| **Патчить после извлечения из того же образа, что на сервере** | Не копировать гигантский `app.js` из другой мажор/минор версии «как в репозитории», пока не сверите тег. |
| **Хранить не только `app.js`, но и способ правки** | Скрипт замены (например `patch_appjs_upload_names.py`) проще отладить при следующем апгрейде Rocket.Chat. |
| **Перед изменениями — бэкап MongoDB** | Миграции Rocket.Chat штатные, но откат без дампа рискован. |

Структура в репозитории:

- **`Patched_file/docker-patch/Dockerfile`** — `FROM …` и `COPY app.js …` под целевой registry/тег.
- **`Patched_file/docker-patch/app.js`** — может быть архивным примером под старый тег; перед сборкой сверять с живым образом.
- **`patch_appjs_upload_names.py`** (в корне `Rocket_chat/`) — транслитерация кириллицы и нормализация имён в **`uploadsOnValidate`** по **`NEEDLE`** (актуально для **8.3.x**); при апгрейде проверить якорь и начало метода в новом бандле.

SSH-алиас, инвентарь хостов: **`README.md`**, **`../docs/servers-inventory.md`**, **`../docs/ssh-and-access.md`**. Пароли и ключи не хранить в документации.

## 2. Порядок работ (на сервере)

Ниже — типовая последовательность. Пути с сервера приведены как они были для одного деплоя; уточняйте `docker compose` и каталог через инвентаризацию.

### Шаг A. Инвентаризация

1. Подключиться по SSH (например алиас **`Rocket_chat`**).
2. Если `docker` требует прав: использовать **`sudo docker`** или пользователя с доступом к сокету.
3. Выполнить:
   - `sudo docker ps -a` — имена контейнеров, образы **с тегами**.
   - Найти compose: часто **`/home/docker/rocketchat/docker-compose.yml`** (рядом **`.env`**, там **`DOMAIN`** и др.).
4. Записать:
   - полный образ Rocket.Chat (**`registry…/rocket.chat:X.Y.Z`** или **`rocketchat/rocket.chat:…`**);
   - сервисы зависимостей (**Mongo**, reverse proxy).

### Шаг B. Резервная копия БД

- Снять дамп **до** остановки приложения или после короткой остановки только **`rocketchat`**, если нужна более тихая БД для экспорта.
- Пример: `mongodump` из контейнера `mongo` в каталог на хосте с датой в имени; проверить размер и наличие коллекций.
- Имеет смысл хранить копию **compose** рядом с бэкапом (например `docker-compose.yml` в ту же метку времени).

### Шаг C. Остановить приложение

- Чаще всего: **`docker compose stop rocketchat`** из каталога со stack — **Mongo не останавливать**, если сразу не делаете обслуживание БД.
- После сборки образа контейнер пересоздают **`compose up --force-recreate`** для сервиса `rocketchat`.

### Шаг D. Извлечь эталонный `app.js` из образа

Бандл в контейнере: **`/app/bundle/programs/server/app/app.js`**.

На хосте (пример):

```bash
sudo docker create --name rc-src <ПОЛНЫЙ_ОБРАЗ_С_ТЕГОМ_КАК_НА_СЕРВЕРЕ>
sudo docker cp rc-src:/app/bundle/programs/server/app/app.js /tmp/rc_app_<версия>.js
sudo docker rm rc-src
```

Дальше копировать файл в каталог сборки, например **`/home/docker/rocketchat/docker-patch-build/app.js`**.

### Шаг E. Применить патч

- **`patch_appjs_upload_names.py`** подменяет **начало **`uploadsOnValidate`** (первый `if (!file.type …)`) и дополнительно вставляет ту же санитизацию **`file.name` после **`Uploads.updateFileComplete`** для **`sendFileMessage`** и у visitor livechat — чтобы **`attachment.title`** в сообщении не был сырным с клиента (иначе русские символы в подписи к файлу на мобильном). Блок таблицы: **транслит кириллицы → латиница**, **пробелы** и недопустимые символы → **`-`**, нижний регистр.
- **`app.js`** для патча должен быть **ванильным** для того же тега образа. **Нельзя** удалять или замещать весь метод `uploadsOnValidate` целиком — после первого `if` должна сохраниться цепочка **`const tmpFile = UploadFS.getTempFilePath`** и далее.
- После сбоя патча — заново **`docker cp` … `app.js`** из официального образа, затем скрипт снова.
- Если уже есть **`const CYR_MAP`** — скрипт выходит без изменений.
- После апгрейда Rocket.Chat править константу **`NEEDLE`** в скрипте под новое начало метода или вставлять блок вручную.
- **Вручную**: вставить эквивалент перед первым **`if (!file.type …`** в `uploadsOnValidate`.

Проверка: подряд идут **транслит-блок**, снова **`if (!file.type …`**, затем **`const tmpFile`**.

### Шаг F. Dockerfile в каталоге сборки

Минимум:

```dockerfile
FROM <тот же образ с registry и тегом, что у сервера>
COPY app.js /app/bundle/programs/server/app/app.js
```

Сборка на сервере из каталога с **`Dockerfile`** и **`app.js`**:

```bash
sudo docker build -t <registry/rocketchat/rocket.chat:X.Y.Z-patched> .
```

Если образ «залип» в слое (**`COPY app.js`**), при изменении **`app.js`** используйте **`docker build --no-cache`**.

Тег **`…-patched`** — согласованное имя «наш образ на базе официального тега».

### Шаг G. Compose и запуск

- В **`docker-compose.yml`** заменить ** `image:`** у сервиса **rocketchat** на собранный тег **`-patched`**.
- Имеет смысл сохранить копию старого compose (например **`docker-compose.yml.bak-<дата>`**).
- Поднять:
  `sudo docker compose up -d rocketchat --force-recreate`

### Шаг H. Проверка

- Логи: **`docker logs rocketchat-…`**, ожидаются **SERVER RUNNING**, миграции без фатальных ошибок (часто «Already at target migration version»).
- HTTP(S): главная страница, **`/api/info`** внутри контейнера по `localhost:3000`.
- Функционально: сценарий, для которого делался патч (например загрузка файла с не-ASCII в имени и проверка сохранённого имени в UI/БД).

## 3. Типичные ошибки

| Симптом | Вероятная причина |
|--------|-------------------|
| Патч «не работает» | В новой версии Rocket.Chat удалили/перенесли хук (как с **`uploadsOnValidate`** между 8.0.2 и 8.3.2); нужен поиск нового места в бандле. |
| Сборка с «старым» `app.js` из репо | В репозитории лежит другой минорный тег; извлекать **`app.js` только из целевого образа.** |
| После патча сломаны превью / sharp / EXIF для изображений | Обрезан целиком метод **`uploadsOnValidate`** или бандл не ванильный — нужно снова взять **`app.js`** из официального образа и патч только по **`NEEDLE`** до строки **`const tmpFile`**. |
| Сборка тянет старый образ при новом **`app.js`** | Запускать **`docker build --no-cache`**. |

## 4. Синхронизация с этим репозиторием

После успешной выкладки имеет смысл:

1. Обновить **`Patched_file/docker-patch/Dockerfile`** под актуальный **`FROM`** (как у сервера).
2. При изменении логики — обновить **`patch_appjs_upload_names.py`** или добавить новый скрипт с понятным именем.
3. Не коммитить огромный **`app.js`**, если команда договорилась держать его только на сервере в **`docker-patch-build/`** — тогда в репо достаточно Dockerfile + скрипта патча + эта инструкция.

## 5. Краткая блок-схема

```
Инвентаризация (тег образа)
        → Бэкап MongoDB
        → Остановка rocketchat
        → docker create/cp эталонного app.js из ТОГО ЖЕ тега
        → патч (скрипт или вручную)
        → Dockerfile + docker build (:…-patched)
        → правка compose + up --force-recreate
        → логи + смоук-тесты
```

Дальнейшие апгрейды Rocket.Chat повторяют тот же цикл: снова **`docker pull`** нужного тега, снова извлечь **`app.js`**, наложить адаптированный патч, пересобрать образ.
